# Kế hoạch Sprint chi tiết theo mô hình Agile
## Dự án: AI Infographic Generator

---

## 1. Mục tiêu tài liệu

Tài liệu này mô tả chi tiết kế hoạch phát triển dự án theo từng sprint, bao gồm phạm vi công việc, các tính năng dự kiến triển khai, kết quả đầu ra mong muốn, rủi ro cần lưu ý và thời gian thực hiện ước tính. Mục tiêu là giúp đội ngũ dự án có một lộ trình rõ ràng để triển khai sản phẩm theo mô hình Agile, đồng thời hỗ trợ Product Owner, Business Analyst, đội phát triển và các bên liên quan có chung kỳ vọng về tiến độ.

Trong tài liệu này, mỗi sprint được giả định kéo dài **2 tuần**. Toàn bộ kế hoạch ban đầu gồm **8 sprint**, tương đương khoảng **16 tuần**, phù hợp cho một giai đoạn từ discovery đến MVP hoàn chỉnh và hardening cơ bản.

---

## 2. Giả định lập kế hoạch

Kế hoạch dưới đây được xây dựng trên một số giả định:

- Nhóm phát triển làm việc theo sprint 2 tuần
- Nhóm có đủ các vai trò cơ bản: Product Owner, BA, Backend, Frontend, AI Engineer, QA
- Dự án ưu tiên hoàn thành MVP trước khi mở rộng tính năng nâng cao
- Hệ thống được triển khai theo pipeline nhiều bước: nhập dữ liệu → extract → summary → spec → render → preview/export
- Các estimate dưới đây là estimate tương đối ở mức planning, có thể điều chỉnh sau mỗi sprint review và retrospective

---

## 3. Tổng quan lộ trình sprint

| Sprint 3 |     2 tuần | AI Understanding Layer          | Sinh summary JSON có cấu trúc                         |

Tổng thời gian dự kiến: **16 tuần**

## 4. Kế hoạch chi tiết từng sprint

# Sprint 1 – Foundation & Project Setup
**Thời gian dự kiến:** 2 tuần  
**Mục tiêu sprint:** Xây dựng nền tảng ban đầu của hệ thống để nhóm có thể bắt đầu phát triển các tính năng cốt lõi trong các sprint tiếp theo.

## 4.1. Tính năng và phạm vi chính

Trong sprint đầu tiên, nhóm chưa tập trung vào AI hay rendering mà ưu tiên tạo ra bộ khung kỹ thuật và nghiệp vụ cho dự án. Đây là sprint tạo “nền móng”, nếu làm không chắc thì các sprint sau sẽ bị chậm và thiếu ổn định.

Các tính năng và hạng mục chính gồm:

- Khởi tạo repository cho frontend và backend
- Thiết lập cấu trúc thư mục dự án
- Thiết lập môi trường local development
- Thiết lập CI/CD cơ bản
- Thiết kế database ban đầu
- Tạo chức năng tạo project mới
- Tạo chức năng upload file cơ bản
- Lưu metadata file và project vào database
- Thiết lập object storage để lưu file upload
- Thiết lập khung authentication nếu có trong MVP
- Hoàn thiện các tài liệu nền tảng còn thiếu như backlog ban đầu, sprint board, definition of done

## 4.2. Công việc chi tiết

### Phân tích và tài liệu
- Rà soát lại PRD, User Stories, Acceptance Criteria
- Chuyển các hạng mục chính sang product backlog
- Tách backlog thành epic, feature, story, task
- Xác định sprint goal và dependency kỹ thuật

### Backend
- Khởi tạo project backend
- Thiết kế schema ban đầu cho bảng user, project, document
- Viết API tạo project
- Viết API upload document metadata
- Tích hợp object storage
- Viết validation cho file input

### Frontend
- Khởi tạo project frontend
- Tạo màn hình tạo project
- Tạo form upload file
- Hiển thị trạng thái project ở mức cơ bản
- Kết nối frontend với API tạo project và upload file

### DevOps
- Thiết lập môi trường development
- Thiết lập pipeline build và deploy cơ bản
- Cấu hình biến môi trường
- Kết nối database và storage

### QA
- Chuẩn bị test case cho create project và upload file
- Kiểm tra luồng tạo project
- Kiểm tra upload file thành công và upload file lỗi

## 4.3. Kết quả mong đợi cuối sprint
- Có thể tạo project mới
- Có thể upload file thành công
- File được lưu vào storage
- Metadata được lưu vào database
- Frontend có thể hiển thị project vừa tạo
- Đội ngũ có môi trường làm việc ổn định

## 4.4. Rủi ro chính
- Chưa thống nhất sớm schema dữ liệu
- Upload file thành công nhưng metadata không đồng nhất
- CI/CD chưa ổn làm chậm các sprint sau

---

# Sprint 2 – Document Ingestion & Extraction
**Thời gian dự kiến:** 2 tuần  
**Mục tiêu sprint:** Hệ thống có thể đọc và trích xuất nội dung từ các tài liệu đầu vào để tạo raw content phục vụ xử lý AI.

## 4.5. Tính năng và phạm vi chính

Sprint này đánh dấu bước đầu tiên của pipeline xử lý nội dung. Trọng tâm không nằm ở AI mà ở việc biến file đầu vào thành dữ liệu thô có thể sử dụng được.

Các tính năng chính gồm:

- Hỗ trợ nhập dữ liệu bằng text trực tiếp
- Đọc file PDF hoặc TXT
- Trích xuất raw text từ tài liệu
- Lưu kết quả extraction vào hệ thống
- Bổ sung trạng thái xử lý project: uploaded, extracting, extracted, failed
- Tạo queue và worker cơ bản cho extraction job

## 4.6. Công việc chi tiết

### Backend
- Viết API nhập text trực tiếp
- Viết service xử lý extraction
- Tạo worker cho document extraction
- Lưu raw extracted text
- Cập nhật trạng thái project theo pipeline
- Ghi log cho quá trình extract

### Frontend
- Bổ sung chế độ nhập text trực tiếp
- Hiển thị trạng thái extracting
- Hiển thị thông báo khi extraction thành công hoặc thất bại

### AI / Data processing
- Chuẩn hóa cấu trúc raw content
- Xác định metadata cần giữ lại như page_count, section headings, file type

### QA
- Test extraction với file hợp lệ
- Test file rỗng, file lỗi, file không hỗ trợ
- Test luồng text input trực tiếp

## 4.7. Kết quả mong đợi cuối sprint
- Hệ thống nhận được file hoặc text
- Worker extract thành công raw content
- Raw content được lưu và truy xuất được
- Trạng thái project phản ánh đúng luồng xử lý

## 4.8. Rủi ro chính
- PDF extraction không ổn định
- Nội dung extract bị lỗi encoding
- Queue setup chưa tốt gây job fail ngầm

---

# Sprint 3 – AI Understanding Layer
**Thời gian dự kiến:** 2 tuần  
**Mục tiêu sprint:** Từ raw content, hệ thống tạo ra summary JSON có cấu trúc và có thể dùng làm đầu vào cho bước tiếp theo.

## 4.9. Tính năng và phạm vi chính

Sprint này là bước đầu tiên tích hợp AI thực sự vào pipeline. Mục tiêu không phải tạo infographic ngay, mà là giúp hệ thống “hiểu” tài liệu.

Các tính năng chính gồm:

- Tích hợp AI text service
- Tạo summary JSON từ raw content
- Trích xuất key points, statistics, entities, tone, topic
- Validate output theo schema
- Lưu summary JSON vào database
- Bổ sung tracking token usage và lỗi AI cơ bản

## 4.10. Công việc chi tiết

### Backend / AI
- Tạo service gọi AI text provider
- Thiết kế schema summary JSON
- Viết validator cho summary JSON
- Tạo logic repair hoặc retry nếu output sai schema
- Lưu usage event cho AI text call
- Cập nhật project state sang understanding hoặc structured bước tiếp theo

### Frontend
- Hiển thị trạng thái “đang phân tích nội dung”
- Hiển thị bản summary cơ bản nếu cần để debug hoặc internal demo

### QA
- Test với nhiều loại tài liệu
- Test summary có đủ trường cần thiết hay không
- Test khi AI timeout, rate limit hoặc trả output lỗi

### BA / Product
- Rà soát chất lượng summary
- So sánh summary với kỳ vọng nghiệp vụ
- Điều chỉnh acceptance criteria nếu cần

## 4.11. Kết quả mong đợi cuối sprint
- Có thể sinh summary JSON hợp lệ từ raw content
- Summary lưu được vào database
- Có thể quan sát trạng thái job AI
- Có dữ liệu trung gian đủ tốt cho bước structuring

## 4.12. Rủi ro chính
- Output AI không ổn định
- Summary thiếu thông tin quan trọng
- Schema validate quá chặt hoặc quá lỏng

---

# Sprint 4 – Structuring Layer
**Thời gian dự kiến:** 2 tuần  
**Mục tiêu sprint:** Từ summary JSON, hệ thống tạo ra infographic specification làm bản thiết kế nội dung cho infographic.

## 4.13. Tính năng và phạm vi chính

Sprint này là bước chuyển từ việc “hiểu” tài liệu sang việc “tổ chức” thông tin theo logic infographic.

Các tính năng chính gồm:

- Tạo infographic spec JSON
- Tổ chức nội dung thành section
- Sinh title, subtitle, layout hint, visual elements
- Xác định theme, canvas ratio, hierarchy
- Validate spec JSON
- Lưu version đầu tiên của spec

## 4.14. Công việc chi tiết

### Backend / AI
- Thiết kế schema spec JSON
- Viết service tạo spec từ summary
- Viết validator cho spec
- Hỗ trợ chọn style hoặc theme cơ bản
- Lưu spec và version vào database
- Gắn spec với project

### Frontend
- Hiển thị trạng thái “đang tổ chức nội dung”
- Xây dựng internal preview đơn giản từ spec JSON
- Hiển thị title, sections, theme được chọn

### QA
- Test spec với nhiều summary khác nhau
- Test trường hợp spec thiếu section hoặc sai canvas ratio
- Test nội dung bị quá dài hoặc không phù hợp layout

## 4.15. Kết quả mong đợi cuối sprint
- Có spec JSON hợp lệ
- Có thể hiển thị preview text-based hoặc block-based từ spec
- Có dữ liệu đủ để gọi bước render trong sprint tiếp theo

## 4.16. Rủi ro chính
- Spec thiếu khả năng biểu diễn trực quan
- Nội dung quá nhiều, khó đưa vào một infographic
- Prompt tạo spec chưa đủ nhất quán

---

# Sprint 5 – Rendering Layer
**Thời gian dự kiến:** 2 tuần  
**Mục tiêu sprint:** Tạo ra infographic preview đầu tiên từ spec JSON.

## 4.17. Tính năng và phạm vi chính

Đây là sprint quan trọng nhất của MVP vì lần đầu tiên sản phẩm tạo ra kết quả trực quan mà người dùng có thể nhìn thấy.

Các tính năng chính gồm:

- Tích hợp AI image service hoặc rendering pipeline
- Chuyển spec JSON thành input cho rendering
- Tạo infographic preview đầu tiên
- Lưu ảnh vào storage
- Hiển thị preview trong frontend
- Lưu image asset metadata

## 4.18. Công việc chi tiết

### Backend / AI
- Tạo service render ảnh
- Kết nối image model provider
- Xử lý kết quả ảnh trả về
- Lưu asset metadata
- Cập nhật project state sang ready nếu render thành công

### Frontend
- Hiển thị preview ảnh
- Bổ sung loading state cho bước render
- Hiển thị lỗi nếu render thất bại

### QA
- Test generate preview với nhiều input
- Test khi provider lỗi hoặc timeout
- Test ảnh lưu đúng URL và hiển thị được

### Product / BA
- Đánh giá chất lượng infographic đầu tiên
- Ghi nhận các lỗi về nội dung, chữ tiếng Việt, layout và hierarchy

## 4.19. Kết quả mong đợi cuối sprint
- Có ít nhất một infographic preview sinh thành công
- Người dùng có thể xem preview trên giao diện
- Project có trạng thái ready ở luồng thành công

## 4.20. Rủi ro chính
- Chất lượng ảnh chưa đạt
- Chữ tiếng Việt sai nhiều
- Chi phí render cao hơn dự kiến

---

# Sprint 6 – Export & Regenerate
**Thời gian dự kiến:** 2 tuần  
**Mục tiêu sprint:** Hoàn thiện luồng MVP cho người dùng, bao gồm tải xuống kết quả và tạo lại infographic.

## 4.21. Tính năng và phạm vi chính

Sau khi đã có preview, sprint này tập trung hoàn thiện trải nghiệm cốt lõi của người dùng.

Các tính năng chính gồm:

- Download infographic
- Regenerate variation mới
- Quản lý nhiều image assets cho cùng một project
- Hiển thị lịch sử variation cơ bản
- Bổ sung giới hạn regenerate nếu cần
- Hoàn thiện luồng end-to-end của MVP

## 4.22. Công việc chi tiết

### Backend
- Viết API download asset
- Viết API regenerate
- Tạo render job mới từ spec đã có
- Lưu nhiều variation vào image_assets
- Thêm usage tracking cho regenerate

### Frontend
- Nút download
- Nút regenerate
- Hiển thị variation list hoặc latest version
- Trạng thái generating variation mới

### QA
- Test download file
- Test regenerate nhiều lần
- Test variation mới không ghi đè asset cũ ngoài ý muốn

### Product / BA
- Nghiệm thu luồng MVP hoàn chỉnh
- Xác định gap còn lại trước hardening

## 4.23. Kết quả mong đợi cuối sprint
- Người dùng có thể tạo infographic, xem preview, tải xuống và regenerate
- MVP end-to-end hoàn chỉnh ở mức khả dụng

## 4.24. Rủi ro chính
- Regenerate gây phát sinh chi phí cao
- Quản lý version asset chưa rõ ràng
- Download URL hoặc quyền truy cập asset chưa an toàn

---

# Sprint 7 – Hardening & Observability
**Thời gian dự kiến:** 2 tuần  
**Mục tiêu sprint:** Nâng cao độ ổn định, quan sát và khả năng vận hành của hệ thống trước khi đưa vào alpha hoặc beta.

## 4.25. Tính năng và phạm vi chính

Sau khi MVP đã chạy được, sprint này tập trung vào chất lượng vận hành thay vì thêm chức năng lớn mới.

Các hạng mục chính gồm:

- Retry policy cho các bước chính
- Better error handling
- Logging chuẩn hóa
- Metrics cơ bản
- Theo dõi cost và usage
- Cải thiện queue handling
- Quản lý failure_reason rõ ràng
- Cải thiện trạng thái project và step history

## 4.26. Công việc chi tiết

### Backend / DevOps
- Chuẩn hóa log format
- Tích hợp metrics
- Bổ sung trace id hoặc correlation id
- Cấu hình retry cho AI và extraction jobs
- Xây dựng dead-letter handling hoặc failure tracking
- Bổ sung project_steps history

### QA
- Test job fail và retry
- Test provider timeout
- Test trạng thái fail và phục hồi
- Test observability data có ghi nhận đúng không

### Product / BA
- Rà soát các case thất bại quan trọng
- Định nghĩa rõ thông báo lỗi người dùng

## 4.27. Kết quả mong đợi cuối sprint
- Hệ thống quan sát được tốt hơn
- Tỷ lệ fail ngầm giảm
- Có khả năng phân tích nguyên nhân lỗi
- Có cơ chế retry và kiểm soát trạng thái tốt hơn

## 4.28. Rủi ro chính
- Logging quá nhiều làm tăng noise
- Retry không kiểm soát làm tăng chi phí
- Truy vết chưa nhất quán giữa API và worker

---

# Sprint 8 – Optimization & Pre-Release
**Thời gian dự kiến:** 2 tuần  
**Mục tiêu sprint:** Tối ưu sản phẩm trước khi phát hành alpha hoặc beta cho người dùng thử nghiệm.

## 4.29. Tính năng và phạm vi chính

Sprint cuối trong kế hoạch này tập trung vào tối ưu chất lượng và khả năng phát hành.

Các hạng mục chính gồm:

- Tối ưu prompt và chất lượng output
- Tối ưu thời gian xử lý
- Cải thiện tiếng Việt trong infographic
- Tối ưu chi phí generate
- Hoàn thiện màn hình lịch sử project
- Rà soát UX của toàn bộ luồng chính
- Chuẩn bị release checklist
- Chuẩn bị tài liệu vận hành cơ bản

## 4.30. Công việc chi tiết

### AI / Backend
- Benchmark prompt và config
- Tối ưu spec để giảm lỗi render
- Xem xét safe mode cho text overlay nếu cần
- Tối ưu chi phí theo usage event

### Frontend
- Cải thiện UI/UX luồng chính
- Tối ưu loading state và feedback message
- Hoàn thiện lịch sử project và asset

### QA
- Regression test toàn bộ MVP
- Kiểm tra hiệu năng cơ bản
- Kiểm tra các case tiếng Việt
- Kiểm tra dữ liệu biên và các case lỗi phổ biến

### Product / BA
- Tổ chức UAT nội bộ
- Thu thập feedback
- Chốt backlog cho giai đoạn alpha/beta tiếp theo

## 4.31. Kết quả mong đợi cuối sprint
- Sản phẩm đủ ổn định để phát hành alpha hoặc beta
- Đã có dữ liệu đánh giá cost, latency, failure rate
- Đội ngũ có căn cứ để quyết định bước phát triển tiếp theo

## 4.32. Rủi ro chính
- Tối ưu quá nhiều làm chậm release
- Thay đổi prompt cuối kỳ gây regression
- Chưa có đủ dữ liệu UAT để chốt quyết định

## 6. Gợi ý phân bổ nguồn lực theo sprint

Để kế hoạch sprint thực tế hơn, có thể hình dung trọng tâm nhân sự theo từng giai đoạn như sau:

- **Sprint 1–2:** Backend, Frontend và DevOps chiếm tỷ trọng lớn
- **Sprint 3–5:** Backend và AI Engineer chiếm tỷ trọng lớn, QA bắt đầu tham gia sâu hơn
- **Sprint 6:** Frontend, Backend và QA phối hợp chặt để hoàn thiện trải nghiệm người dùng
- **Sprint 7–8:** Backend, QA, DevOps và Product cùng tập trung vào hardening và release readiness

## 7. Lưu ý khi triển khai thực tế

Kế hoạch sprint trên là kế hoạch ban đầu. Trong Agile, sprint plan không phải cam kết cứng cho toàn bộ 16 tuần, mà là một bản định hướng có kiểm soát. Sau mỗi sprint, nhóm cần xem xét lại:

- Velocity thực tế của nhóm
- Độ khó kỹ thuật phát sinh
- Chất lượng output AI
- Phản hồi từ stakeholder
- Thay đổi trong ưu tiên sản phẩm

Nếu một số hạng mục mất nhiều effort hơn dự kiến, nhóm có thể tách nhỏ backlog, dời một phần sang sprint sau hoặc điều chỉnh lại sprint goal, thay vì cố nhồi toàn bộ scope và làm giảm chất lượng.

Thay vì cố gắng xây dựng toàn bộ sản phẩm trong một lần, nhóm phát triển theo từng lát cắt giá trị: từ nền tảng hệ thống, xử lý dữ liệu đầu vào, hiểu nội dung, tổ chức cấu trúc, tạo infographic, đến hoàn thiện trải nghiệm và tối ưu vận hành.
